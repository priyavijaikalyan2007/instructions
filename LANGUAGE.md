# Language and Communication Guidelines

All agents, including coding and review agents, must adhere to these language guidelines for all natural language output, including code comments, documentation (code, technical, and user-facing), and general communication.

## Core Principles

*   **Neutral and Unemotional:** Maintain a calm and professional tone. Avoid emotive language or subjective intensifiers.
*   **Unbiased and Objective:** Present information based on facts and evidence. Avoid personal opinions or assumptions.
*   **Precise:** Use clear and specific language to convey meaning accurately.
*   **Upleveled but Simple:** Use professional business language that remains easy to understand.
*   **Polite:** Use respectful and courteous phrasing.
*   **Standard British English:** Use British spelling and grammar conventions; however, use American spellings (e.g., 'optimize' instead of 'optimise', 'color' instead of 'colour', 'honor' instead of 'honour').

## Accessibility and Clarity

*   **Avoid Technical Jargon:** Use plain English whenever possible. If a technical term is necessary, provide a brief, simple explanation.
*   **No Profanity:** Swear words or offensive language are strictly prohibited.
*   **Simple Vocabulary:** Avoid unnecessarily complex or obscure words. The goal is to be accessible to everyone, including non-native English speakers.
*   **Concise Sentences:** Use short, direct sentences to improve readability.

## Application

These guidelines apply to:
*   In-code comments and documentation (e.g., JSDoc, Docstrings).
*   README files and technical specifications.
*   User manuals and help documentation.
*   Commit messages and pull request descriptions.
*   All interactions with users and other agents.
